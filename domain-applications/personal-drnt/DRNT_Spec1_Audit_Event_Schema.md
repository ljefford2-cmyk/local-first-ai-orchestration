# DRNT Interface Specification

## Spec 1: Audit/Event Schema

**FINAL** · March 2026 · `schema_version: "1.5"` · Build Target: Docker + JSONL on WSL2/Ubuntu

---

## 1. Purpose

This spec defines the single event format for the DRNT append-only audit log. Every event in the system—routing decisions, WAL state changes, Context Packager actions, model responses, human decisions, overrides, anomalies, and failures—has one place to go and one shape when it gets there. The audit log is the foundation. Specs 2–5 all write to this schema.

## 2. Storage Format

### 2.1 File Format and Location

JSONL (JSON Lines). One JSON object per line. One file per UTC calendar day:

```
drnt-audit-2026-03-21.jsonl
```

Location on the Desktop hub (WSL2/Ubuntu), inside a Docker volume:

```
/var/drnt/audit/
```

Bind-mounted from host filesystem so logs survive container restarts. Only the log writer container has write access. All other containers—including the orchestrator—mount this path read-only. The orchestrator does not write directly to audit files under any circumstances; it sends events to the log writer over a Unix domain socket (Section 7).

### 2.2 Integrity Chain

Every event includes a SHA-256 hash of the previous event's JSON line (the `prev_hash` field). The first event of each day file chains to the last event of the previous day's file. This creates a tamper-evident chain: modifying any event invalidates all subsequent hashes. A standalone script (`drnt-audit-verify`) walks the chain and reports breaks. It runs nightly via cron and on-demand, independent of the orchestrator.

### 2.3 UTC Midnight Rollover

The log writer determines the target file by the UTC date of the `committed_at` timestamp. At day boundary: (1) finish current event to current file, (2) record SHA-256 hash of the last line, (3) next event opens a new file chaining from that hash. The log writer is single-threaded with respect to writes—no race condition.

### 2.4 Genesis Seed

The first event ever written uses a seed hash. SHA-256 of the literal string `"DRNT-GENESIS"` (UTF-8, no trailing newline):

```
9cd9da92046a4de28d7cec92f9559e542ba6a52af9fe44513bd26b0c2838a4d4
```

Each new daily file chains from the last event of the previous day. If no previous day exists, the genesis seed is used.

### 2.5 Guarantees and Limitations

The chain guarantees tamper evidence: if any event is modified, inserted, or deleted, the chain breaks. It does not prevent tampering. For v1 on a single-user personal system, this is acceptable. The chain provides a foundation for stronger guarantees (signed hashes, remote attestation) if DRNT moves beyond personal use.

## 3. Event Envelope

Every event, regardless of type, carries these fields. No exceptions.

| Field | Type | Description |
| --- | --- | --- |
| `schema_version` | string | `"1.5"`. Consumers must check before parsing. |
| `event_id` | UUIDv7 | Assigned by log writer at commit time. Identity of the log record, not the logical event. |
| `source_event_id` | UUIDv7 | Generated by emitter at occurrence time. Identity of the logical event. Stable across retransmissions. Used for deduplication (Section 7.2). |
| `timestamp` | ISO 8601 µs | UTC. When the event happened. Set by emitter. |
| `committed_at` | ISO 8601 µs | UTC. When the log writer durably wrote it. The gap is delivery latency. |
| `event_type` | string enum | Dot-namespaced. Determines payload structure. |
| `job_id` | UUIDv7 │ null | The job this event belongs to, or null for system events. |
| `parent_job_id` | UUIDv7 │ null | If this job was spawned by a redirect or escalate override, points to the original job. Null for user-initiated jobs. |
| `capability_id` | string │ null | Governing capability for this job. Null for system events and `job.submitted` (pre-classification). Set at `job.classified` and carried through all subsequent events. Never mutated. |
| `wal_level` | int (-1..3) │ null | Effective WAL of governing capability at event time. -1 = suspended. Null when `capability_id` is null. |
| `source` | string enum | `orchestrator` │ `context_packager` │ `egress_gateway` │ `watch_app` │ `phone_app` │ `human` │ `system` │ `log_writer`. |
| `prev_hash` | hex64 | SHA-256 of previous event's complete JSON line (UTF-8, no trailing newline). |
| `durability` | string enum | `"durable"` or `"best_effort"`. Set by emitter per event type. |
| `payload` | object | Event-type-specific data. Fixed per `event_type`. All fields present in every instance; null for inapplicable values. |

### 3.1 Capability Scoping: One Job, One Governing Capability

In v1, every job has exactly one governing capability, established at `job.classified` and carried in the envelope through all subsequent events. Classification is a system action—it runs before capability permission checks and determines which gates apply.

**Auxiliary capabilities** emit their own `wal.permission_check` events with their own `capability_id` in the payload. The envelope always reflects the governing capability. Auxiliary checks carry a `dependency_mode` (`required` | `optional` | `best_effort`) that determines block/skip/log-and-continue behavior (Spec 2 Section 2.1).

**Failure attribution:** failures are attributed to the capability that failed, not the governing capability. `job.failed` carries `failing_capability_id`. `context.strip_detail` separates `detected_by` (who caught it) from `failing_capability_id` (who failed). Sentinel values (`"egress_config"`, `"egress_connectivity"`) are excluded from WAL failure counters.

**Successor jobs:** redirect and escalate overrides terminate the original job and spawn a successor with a new `job_id` and `parent_job_id` linking to the original. The invariant holds: the governing capability is never mutated within a job.


## 4. Event Types

Dot-namespaced. Payload schema fixed per type. All fields present in every instance; null for inapplicable values, not omission.

### 4.1 Job Lifecycle

Every job produces at minimum `job.submitted` and one terminal event: `job.delivered`, `job.failed`, or `job.revoked`. Retry sequences: `job.failed` may be followed by a new `job.dispatched` with incremented `retry_sequence` (v1: same capability only). Redirect and escalate overrides terminate the original job and spawn a successor with a new `job_id`.

| event_type | D | Payload Fields |
| --- | --- | --- |
| `job.submitted` | D | `raw_input` (string), `input_modality` (voice│text│tap), `device` (watch│phone), `request_category` (string│null — null before classification) |
| `job.classified` | D | `request_category` (string), `confidence` (float 0–1), `local_capable` (bool), `routing_recommendation` (local│cloud│multi), `candidate_models` (string[]), `governing_capability_id` (string), `declared_pipeline` (array of {capability_id: string, dependency_mode: required│optional│best_effort} — snapshot of the governing capability's pipeline at classification time) |
| `job.queued` | BE | `reason` (connectivity│rate_limit│capacity), `position` (int), `estimated_wait_ms` (int│null) |
| `job.dispatched` | D | `target_model` (string), `route_id` (string — Spec 4 egress registry ref; never contains secrets), `context_package_id` (string), `wal_permission_check_ref` (string — source_event_id of authorizing check), `retry_sequence` (int — 0-indexed), `same_capability_retry` (bool), `assembled_payload_hash` (string SHA-256 — matches context.packaged), `egress_config_hash` (string SHA-256 — hash of egress.json as loaded by gateway). Emitted when gateway confirms outbound transmission, BEFORE model response. |
| `job.response_received` | D | `model` (string), `latency_ms` (int), `token_count_in` (int), `token_count_out` (int), `cost_estimate_usd` (float│null), `truncated` (bool), `result_id` (string UUIDv7 — pointer to results store), `response_hash` (string SHA-256 — see Section 4.4.1) |
| `job.delivered` | D | `delivery_target` (watch│phone), `display_format` (string), `acknowledged` (bool), `review_gate_applied` (none│pre_delivery│post_action — the gate that governed this delivery) |
| `job.failed` | D | `error_class` (timeout│rate_limit│model_error│context_error│internal), `error_message` (string), `retryable` (bool), `retry_count` (int), `failing_capability_id` (string — the capability whose action failed; may differ from envelope; sentinel values `"egress_config"`/`"egress_connectivity"` for transport failures, excluded from WAL counters) |
| `job.revoked` | D | `reason` (user_cancel│escalation_supersede│offline_retroactive), `override_source_event_id` (string — source_event_id of the triggering human.override), `successor_job_id` (string│null — replacement job if redirect/escalate; null for pure cancellation), `memory_objects_superseded` (string[] — IDs of memory objects marked superseded) |

**Three terminal states:** `job.delivered` (execution succeeded, result presented to user), `job.failed` (execution did not complete, or pre-delivery cancel/redirect), `job.revoked` (execution completed and delivered, but human subsequently withdrew acceptance). Revocation does not delete prior events—`job.delivered` and `model.response` remain in the log.

### 4.2 Context Packager Events

Every outbound cloud dispatch produces exactly one `context.packaged` event. `context.strip_detail` is emitted once per non-pass transformation.

| event_type | D | Payload Fields |
| --- | --- | --- |
| `context.packaged` | D | `context_package_id` (UUIDv7), `job_id`, `fields_included` (field_id[]), `fields_stripped` (field_id[]), `fields_generalized` (field_id[]), `exclusion_summary` (array of {source_id, reason: local_only│below_confidence│over_budget}), `memory_objects_attached` (int), `memory_retrieval_ms` (int), `outbound_token_estimate` (int), `assembled_payload_hash` (SHA-256 of the prompt as stored in context store), `context_object_hash` (SHA-256 of the full context object JSON as stored) |
| `context.strip_detail` | D | `context_package_id` (string), `field_id` (UUIDv7 — ref to eligible_context_fields entry), `field_name` (string — human-readable, for grep), `span` ({start_char: int, end_char: int}│null — for sub-field transforms in mixed content), `original_class` (name│location│date│financial│credential│custom│unclassifiable), `classification_status` (matched│safe_none│unclassifiable), `action` (stripped│generalized), `method` (redact│mask│coarsen│omit│default_deny), `detected_by` (packager│egress_gateway), `failing_capability_id` (string│null — `"context.package"` for egress catches; null for routine strips and packager-side default-deny), `reason` (string) |

**Rollup arrays use field_ids, not names.** A single `field_id` may appear in multiple rollup arrays when a mixed-content field is partly passed and partly transformed. The arrays are not mutually exclusive. Strip_detail events carry `span` for sub-field precision.

**Default-deny attribution:** when the Packager blocks unclassifiable content, `failing_capability_id` is null. This is conservative success. Only egress gateway catches (where the Packager should have caught it but didn't) set `failing_capability_id: "context.package"` and trigger strip-failure trust consequences.

### 4.3 WAL State Events

| event_type | D | Payload Fields |
| --- | --- | --- |
| `wal.promoted` | D | `capability_id` (string), `capability_name` (string), `from_level` (int -1..3), `to_level` (int 0..3), `criteria_met` (object — snapshot of evidence), `approved_by` (string: human) |
| `wal.demoted` | D | `capability_id` (string), `capability_name` (string), `from_level` (int -1..3), `to_level` (int -1..3), `trigger` (error│anomaly│override│model_change│manual│config_reconcile│strip_failure), `incident_ref` (string│null — source_event_id of triggering incident) |
| `wal.permission_check` | D | `capability_id` (string — the specific capability checked; may differ from envelope for auxiliary checks), `job_id` (string), `requested_action` (string — canonical action from Spec 2 Section 3), `current_level` (int), `required_level` (int), `result` (allowed│blocked│held), `delivery_hold` (bool — true when result is allowed but deliver_result must apply pre_delivery gate; false otherwise), `block_reason` (string│null), `hold_type` (pre_action│cost_approval│on_accept│null — present only when result is held), `dependency_mode` (required│optional│best_effort│null — null for governing capability checks) |

**Three-state permission result:** `allowed` (action proceeds), `blocked` (capability's WAL level does not permit—architectural denial), `held` (action blocked pending human input). `delivery_hold: true` on an allowed result means the dispatch proceeds but the downstream `deliver_result` step applies a `pre_delivery` gate. This is the canonical encoding—`pre_delivery` is never a held result.

### 4.4 Model Response Events

Every model call produces a `model.response` event. Single-model: one event, `selected: true`. Multi-model: one per model, `selected: true` on the delivered response.

| event_type | D | Payload Fields |
| --- | --- | --- |
| `model.response` | D | `job_id` (string), `model` (string), `model_version` (string│null), `latency_ms` (int), `token_count_in` (int), `token_count_out` (int), `cost_estimate_usd` (float│null), `result_id` (string UUIDv7 — pointer to results store), `response_hash` (string SHA-256), `selected` (bool) |
| `model.error` | D | `job_id` (string), `model` (string), `error_type` (timeout│rate_limit│auth│server_error│malformed_response), `http_status` (int│null), `retry_attempted` (bool), `fallback_model` (string│null) |

#### 4.4.1 Response Hash Definition

The `response_hash` is the SHA-256 hex digest of the exact bytes stored in the results store for that `result_id`. For text responses: UTF-8 encoding as received, no normalization. For structured/multimodal: UTF-8 of canonical JSON serialization (keys sorted alphabetically, no trailing whitespace, no pretty-printing). The `result_id` is the join key between the audit log and the results store.

### 4.5 Human Decision Events

| `human.reviewed` | D | `job_id` (string), `decision` (accepted│modified│rejected│resubmitted), `device` (watch│phone), `review_latency_ms` (int), `modification_summary` (string│null), `modified_result_id` (UUIDv7│null — pointer to human-edited result in results store; present only for modified), `modified_result_hash` (SHA-256│null — hash of modified result; present only for modified), `derived_from_result_id` (UUIDv7│null — the original model-produced result_id that was modified) |

**Modified result lineage:** `model.response.result_id` (original) → `human.reviewed.derived_from_result_id` (same) → `human.reviewed.modified_result_id` (new artifact). Both exist in the results store. Both hashes are in the audit log.

## 5. Hash Chain Mechanics

The `prev_hash` of event N is the SHA-256 hex digest of the complete JSON line of event N−1, encoded as UTF-8, excluding the trailing newline character. The genesis seed (Section 2.4) is used for the first event ever and when no previous day's file exists.

## 6. Retention and Querying

### 6.1 Retention

V1: retain all log files indefinitely. Disk space is not a constraint at personal scale.

### 6.2 Querying

V1 tooling: `jq`, `grep`, `tail -f`. No database. SQLite indexes can layer on later without changing the log format.

```bash
# All events for a job:
jq 'select(.job_id == "019508a3...")' drnt-audit-2026-03-21.jsonl

# Strip failures (egress catches with attribution):
jq 'select(.event_type == "context.strip_detail"
    and .payload.failing_capability_id != null)' drnt-audit-*.jsonl

# Permission checks with delivery holds:
jq 'select(.event_type == "wal.permission_check"
    and .payload.delivery_hold == true)' drnt-audit-*.jsonl

# Held permission checks (action blocked):
jq 'select(.event_type == "wal.permission_check"
    and .payload.result == "held")' drnt-audit-*.jsonl

# Successor chain for redirected/escalated jobs:
jq 'select(.parent_job_id == "...")' drnt-audit-*.jsonl

# All WAL demotions:
jq 'select(.event_type == "wal.demoted")' drnt-audit-*.jsonl

# Replay a job's pipeline:
jq 'select(.job_id == "..." and .event_type == "job.classified")
    | .payload.declared_pipeline' drnt-audit-*.jsonl

# Live monitoring:
tail -f drnt-audit-2026-03-21.jsonl | jq '.'
```

### 6.3 Results Store Interface

The results store is separate from the audit log. Join key: `result_id` (UUIDv7). The store preserves the exact bytes hashed in `response_hash`. Modified results are stored under `modified_result_id` as separate immutable artifacts. Both original and modified are verifiable via their respective hashes.

### 6.4 Context Store Interface

The context store is separate from the audit log. Location: `/var/drnt/context/`. Join key: `context_package_id`. Each package is a JSON file containing the full context object (`retrieved_candidates`, `eligible_context_fields`, `assembled_payload`). Written once, never modified. Verifiable: `assembled_payload_hash` anchors the payload; `context_object_hash` anchors the full object.

## 7. Log Writer Architecture

### 7.1 Process Design

The log writer is a separate lightweight process. It listens on a Unix domain socket, receives serialized event objects, deduplicates on `source_event_id` (Section 7.2), assigns `event_id` (UUIDv7 at commit time) and `committed_at`, computes `prev_hash`, serializes to JSON, and appends. On startup it reads the last line of the current/previous day's file to initialize chain state, or uses the genesis seed.

### 7.2 Deduplication

In-memory set of `source_event_id` values from the current and previous day's files. Duplicates are dropped silently. This handles device replay and socket retries without corrupting the chain.

### 7.3 Durability Classes and ACK Semantics

| Class | Write Semantics | ACK Behavior |
| --- | --- | --- |
| **durable** | Append + fsync, then return ACK to emitter. | Emitter blocks until ACK. 5-second timeout, then retry (same source_event_id). Gated action does not proceed until ACK confirmed. |
| **best_effort** | Append to write buffer. Flush on next durable write or 1-second interval. | Fire-and-forget. Action proceeds immediately. |

**Fail-closed guarantee:** if the log writer is down, durable events cannot be ACK'd, so the orchestrator cannot dispatch, change WAL state, or process overrides. The system halts. Best-effort events are silently dropped and associated actions proceed normally.

### 7.4 Durability by Event Type

| Durable (write-before-proceed) | Best-Effort (fire-and-forget) |
| --- | --- |
| `job.submitted`, `job.classified`, `job.dispatched`, `job.response_received`, `job.delivered`, `job.failed`, `job.revoked`, `context.packaged`, `context.strip_detail`, `wal.promoted`, `wal.demoted`, `wal.permission_check`, `model.response`, `model.error`, `human.reviewed`, `human.override`, `system.startup`, `system.shutdown`, `system.integrity_check`, `system.config_change`, `system.model_change`, `egress.validation_failure` | `job.queued`, `human.feedback`, `system.health_check`, `system.connectivity` |

### 7.5 Event Emission from Components

Components provide all fields except `event_id`, `committed_at`, and `prev_hash`. They provide `source_event_id` (self-generated UUIDv7), `timestamp` (at occurrence), and `durability` (per event type).

**Write path:** Component → Unix socket → Log Writer → dedup → assign event_id + committed_at + prev_hash → JSONL file → fsync if durable → ACK.

### 7.6 Watch/Phone Events

Transmitted to the desktop hub over Tailscale. If offline, events queue locally on the device and flush on reconnection. `source_event_id` and `timestamp` are set on-device; `committed_at` is set by the log writer. The gap between `timestamp` and `committed_at` is the offline queue duration.
